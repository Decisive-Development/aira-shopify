# CLAUDE.md — aira Shopify theme

Read every session. This is law. It governs code, structure, and process.
Copy rules load automatically from `.claude/rules/copy.md` when you edit templates; the full copy and claims law is `AIRA_Brand_Copy_Standards.md` (repo root), SEO is `AIRA_SEO_Meta_Plan.md`. On wording the copy doc wins; on SEO the SEO plan wins; on code this file wins. Deviations need Tom's sign-off.

## 1. What this is

aira: premium magnetic nasal strips, GCC market, launching ~late July 2026. Custom Shopify theme (Online Store 2.0): Liquid + Tailwind v4 (compiled) + CSS custom properties + vanilla JS with Lenis for smooth scroll. Pre-revenue and bootstrapped, so keep weight, dependencies, and section count low. Every addition is a liability you maintain.

**Settled, do not reopen:** UK English throughout customer-facing copy. An Arabic version of the site is a future, funded project, not now, so don't add half-translated pages.

## 2. Stack and structure

- **Stack:** Shopify OS 2.0 (JSON templates), Liquid, Tailwind v4 compiled via `@tailwindcss/cli`, vanilla JS plus Lenis (smooth scroll). Reviews via the Judge.me app.
- **No React, no GSAP, no Alpine, no new JS frameworks.** New interactivity is vanilla JS: progressive enhancement (page works without JS), small per-section scripts, no inline script soup. Lenis is the only JS library, gated behind `prefers-reduced-motion`.
- **Folders:** `sections/` (~72), `snippets/`, `templates/` (mostly JSON), `assets/`, `config/`, `layout/`, `locales/`.
- **Canonical shared snippets, reuse these, never reinvent:** `cta-button` (all buttons, pass `variant:`), `cta-banner` (CTA bands), `section-heading` (eyebrow + heading), `image`, `product-card`.
- **Naming:** lowercase-kebab, describing function not page or order (`reviews-grid`, never `homepage-thing-2` or a reversed-word twin of a file that already exists).
- **Environment:** store `aira-breathing`, dev theme `Aira-theme`, repo `Decisive-Development/aira-shopify`, work on `main`.

## 3. Commands

- Build CSS: `npm run build:css`  (compiles `assets/critical.source.css` → `assets/critical.css`)
- Watch CSS: `npm run watch:css`
- Local dev: `npm run dev`  (`shopify theme dev`, store `aira-breathing`)
- Lint: `shopify theme check`
- Sync theme: `shopify theme pull` / `shopify theme push`

## 4. Build pipeline and generated files

**IMPORTANT — YOU MUST edit `assets/critical.source.css`, never `assets/critical.css`.** The latter is generated by `npm run build:css`; edits to it are wiped on the next build. Commit the source and the rebuilt CSS together.

Generated or app-managed, never hand-edit:

- `assets/critical.css` (Tailwind output)
- `config/settings_data.json` (the Shopify admin writes this; it says so in its header)
- `node_modules/` (npm)
- Judge.me app blocks inside templates and section settings

## 5. Design tokens (use these, never raw values)

Every colour, space, and type size comes from a token. No raw hex or magic px in a section, ever.

**Colours** are set in the **Brand colours** settings group (`config/settings_schema.json`), the canonical source, and flow to the `--color-*` custom properties; the utility classes (`bg-main`, `text-secondary`, `bg-highlight`…) read from them. Change colours there.
<!-- Maintainer note: wiring the settings panel to actually drive the CSS vars is a pending task; today the values are also hard-set in critical.source.css. -->

- CTA / accent `--color-highlight` #FF4000 · navy `--color-main` #0A1A34 · cream `--color-secondary` #F1EBE1
- support: `--color-tertiary-light` #9BB7D4 · `--color-cream-light` rgb(248,245,240) · `--color-highlight-green` #28965A · `--color-highlight-blue` #087CA7 · `--color-title` #2A3142 · `--color-copy` #5C6470
- backgrounds: `--color-bg` #FFFFFF · `--color-bg-soft` #FAFAFA · `--color-bg-muted` #F2F2F2

**Type:** Satoshi for headings (class `font-heading`, `--font-heading`), Nunito Sans for body (class `font-body`, `--font-body`), both self-hosted. Tailwind default size scale (`text-xs`–`text-7xl`).
**Radius:** Tailwind scale; button baseline `rounded-xl` = 0.75rem. Inputs use `settings.input_corner_radius`.
**Spacing:** Tailwind base `--spacing: 0.25rem`. Max page width `settings.max_page_width` (90rem default).

Anything used in the theme that isn't a token above is drift, flag it.

## 6. Hard rules (code)

1. **IMPORTANT — YOU MUST NOT duplicate or fork a file.** No `section-v2.liquid`, no parallel "fix" file, no reversed-word twin (`science-nasal-breathing` vs `nasal-breathing-science`). Fix in place; refactor the shared thing.
2. **Never add a section when one already does the job.** Check the existing ~72 first; reuse or extend.
3. **Everything from tokens (§5).** No hardcoded colours, spacing, or type.
4. **Repeated markup becomes a snippet** rendered with `{% render %}` (never `{% include %}`); one canonical snippet per UI element (buttons are always `cta-button`, variants are parameters).
5. **Section content is editable via `{% schema %}` settings,** not hardcoded in Liquid.
6. **Dead code is deleted, not commented out.** Git is the history.

Copy rules live in `.claude/rules/copy.md` (auto-loads when you edit templates) and `AIRA_Brand_Copy_Standards.md`.

**Before you touch anything:** read the file top to bottom; grep for the class, token, snippet, or section you're about to add and reuse or extend it; state what you'll change and every section it could affect; if unclear, stop and ask Tom (guessing creates the V2/V3 loop).

**Accessibility and performance (non-negotiable):** semantic HTML, alt text, input labels, visible focus, AA contrast; no layout shift; lazy-load below-the-fold images; no render-blocking third-party scripts; animation respects `prefers-reduced-motion` and stays restrained (no bounce, pulse, scale).

## 7. Good example (copy this pattern)

```liquid
{%- comment -%}
  A section is a thin, schema-driven wrapper that renders shared snippets and
  uses token utility classes. No hardcoded hex, no copy-pasted markup.
{%- endcomment -%}
<section class="bg-main text-secondary">
  <div class="mx-auto px-4">
    {%- render 'cta-banner',
      eyebrow: section.settings.eyebrow,
      heading: section.settings.heading,
      primary_label: section.settings.primary_label,
      primary_url: section.settings.primary_url,
      primary_variant: 'shop'
    -%}
  </div>
</section>
{% schema %}
{ "name": "...", "settings": [ ... ] }
{% endschema %}
```

## 8. Workflow and Definition of Done

- **Briefs:** Tom delivers work as `Brief.txt` or `AIRA_*_Brief_VSC.txt`. Execute the brief exactly; copy strings in briefs are final and pre-approved unless marked as a dependency.
- **Footer nav** comes from Shopify Navigation menus, not hardcoded.
- **Commits:** small, single-purpose, imperative messages ("Add reviews pagination", not "save"). Commit straight to `main`. Never force-push.
- **Gate:** `shopify theme check` runs in CI on every push to `main`, and the Tailwind build must succeed. Don't leave `main` red.

**Definition of Done (self-verify before you say it's done):**
- [ ] No duplication; reused existing snippets, sections, and tokens.
- [ ] No hardcoded colours, spacing, or type.
- [ ] If CSS changed: edited `critical.source.css`, rebuilt, committed both.
- [ ] `shopify theme check` passes with zero new offences.
- [ ] Renders on mobile and desktop; no layout shift; text links stay text links on mobile.
- [ ] Copy obeys the copy rule and standards doc (UK spelling, no dashes or arrows, no uncited stats).
- [ ] Changed only what the brief asked; listed every shared element touched and the sections affected.
- [ ] Stated what was verified and how.

**Safety:** never delete customer or store data, change theme or store settings, or alter access. Never invent a product claim, number, review, or design token. When unsure, stop and ask Tom.
